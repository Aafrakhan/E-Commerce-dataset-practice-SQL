Good call — that's the standard approach for a live system. Since the inserts already specify explicit IDs, you can keep AUTO_INCREMENT and still insert those values; MySQL will just pick up counting from where you left off for future inserts.

I went ahead and applied the same logic to product_id and order_id too, since in a real e-commerce system new products and new orders also need auto-generated IDs.

All three primary keys (customer_id, product_id, order_id) are now AUTO_INCREMENT.

Since your INSERT statements already specify explicit IDs (1, 2, 3...), MySQL will load them as-is and automatically set its internal counter to continue from the highest existing ID. So:

  • After loading, the next customer you insert (without specifying an ID) will get customer_id = 61
  • Same logic applies to new products and new orders

Orchestrated SQL-based answer key compilation

Here's the answer key for all 15 questions.

Basic SELECT & Filtering

  1. Customers from Mumbai
sql
SELECT * FROM customers
WHERE city = 'Mumbai';
  1. Electronics products priced above ₹2000
sql
SELECT * FROM products
WHERE category = 'Electronics' AND price > 2000;
  1. Orders that were Cancelled or Returned
sql
SELECT * FROM orders
WHERE order_status IN ('Cancelled', 'Returned');
  1. Female customers who signed up after Jan 1, 2026
sql
SELECT customer_name, email FROM customers
WHERE gender = 'Female' AND signup_date > '2026-01-01';
  1. Orders paid via UPI
sql
SELECT * FROM orders
WHERE payment_method = 'UPI';

Sorting & Limiting

  1. Top 5 most expensive products
sql
SELECT product_name, category, price FROM products
ORDER BY price DESC
LIMIT 5;
  1. 10 most recent orders
sql
SELECT * FROM orders
ORDER BY order_date DESC
LIMIT 10;

Aggregations

  1. Total revenue from Delivered orders
sql
SELECT SUM(total_amount) AS delivered_revenue
FROM orders
WHERE order_status = 'Delivered';
  1. Order count by payment method
sql
SELECT payment_method, COUNT(*) AS total_orders
FROM orders
GROUP BY payment_method
ORDER BY total_orders DESC;
  1. Average order value across all orders
sql
SELECT AVG(total_amount) AS avg_order_value
FROM orders;
  1. Category with highest total stock_quantity
sql
SELECT category, SUM(stock_quantity) AS total_stock
FROM products
GROUP BY category
ORDER BY total_stock DESC
LIMIT 1;

Joins

  1. Customer name + product name for every order
sql
SELECT c.customer_name, p.product_name, o.quantity, o.order_date
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
JOIN products p ON o.product_id = p.product_id;
  1. Total amount spent by each customer
sql
SELECT c.customer_name, SUM(o.total_amount) AS total_spent
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
GROUP BY c.customer_id, c.customer_name
ORDER BY total_spent DESC;
  1. Products that have never been ordered
sql
SELECT p.product_id, p.product_name
FROM products p
LEFT JOIN orders o ON p.product_id = o.product_id
WHERE o.order_id IS NULL;

Business-style

  1. City with highest total order revenue
sql
SELECT c.city, SUM(o.total_amount) AS total_revenue
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
GROUP BY c.city
ORDER BY total_revenue DESC
LIMIT 1;

A few notes as you go through these:

  • Q9, Q11, Q13, Q15 all use GROUP BY — make sure you understand why every non-aggregated column in SELECT must appear in GROUP BY.
  • Q14 (finding unordered products) is a very common interview trick question — the LEFT JOIN ... WHERE ... IS NULL pattern comes up a lot, so it's worth practicing variations of it.
